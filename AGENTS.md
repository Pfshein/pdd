# AGENTS.md — правила для моделей, работающих в PDD

Этот файл авто-подгружается агентами (Claude Code / Codex / совместимые) как контекст
репозитория. Держи его коротким: это правила и указатели, а не документация. Архитектура —
в [orchestrator/README.md](orchestrator/README.md); бэклог задач — в
[docs/LOOP_ENGINEERING_PROJECT.md](docs/LOOP_ENGINEERING_PROJECT.md).

PDD — детерминированный Loop Engineering-рантайм: код прогоняет задачу по графу стадий,
LLM исполняет роль внутри стадии, маршрут принимает `router`, не модель.

## Golden rules (правки кода)

- Делай **минимальное** изменение, решающее задачу. Не рефактори несвязанное.
- Простые функции и данные, без ООП — в стиле текущего кода.
- Не меняй публичное поведение CLI, если задача явно об этом не просит.
- Сохраняй существующие артефакты и форматы файлов, если не указана миграция.
- Добавляй фокусные тесты на любую новую логику routing / persistence / CLI / report.
- **Не ослабляй sandbox-дефолты** (`--cap-drop ALL`, `--read-only`, non-root,
  no-new-privileges, `--network none` для `TEST_RUN`) без задачи, явно про sandbox.
- **Секреты — только через env.** Никогда в argv / артефакты / логи / отчёты / тесты.
- Тесты ядра **не должны** требовать реальных model / network / Docker / Jira / GitHub —
  используй stubs (`stubs/qwen_stub.py`, monkeypatch).
- Видишь несвязанные грязные файлы — игнорируй их.

## Definition of Done

Задача готова, только когда: поведение реализовано; добавлены/обновлены фокусные тесты;
обновлён CLI/help/report-текст, если менялся пользовательский контракт; ошибки
детерминированы и читаемы; новые артефакты задокументированы; ядро тестируется без реальных
model/network/Docker.

## Dev workflow

- Ветка на задачу: `PDD-XX-short-name` (для backlog-карточек) или описательное имя
  (`docs-...`, `fix-...`). Один маленький PR на задачу.
- Прогон: `python -m pytest -q` (или `pdd doctor` для self-check окружения).
- Коммит-сообщение через файл (`git commit -F`); окончание:
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`. Автор — Pfshein.
- **Грабли:** конфликт мёржа в `cli.py build_parser` однажды съел сабкоманды — при мёрже
  веток, трогающих CLI, проверяй, что все сабкоманды на месте.

## Документация: конвенция

- **`README.md` в каждой значимой директории** (co-location): человекочитаемый reference,
  авто-рендерится на GitHub. Канон архитектуры один — `orchestrator/README.md`; остальные
  ссылаются, не дублируют.
- Документируй **медленно меняющееся**: назначение, контракты, инварианты, грабли — не
  построчное поведение (его описывают код и тесты).
- Не плоди доки: директории из очевидных данных (`schemas/`) — 3–5 строк.
- Time-bound доки (`docs/STATUS.md`) явно помечай как snapshot.

Шаблон секций для per-directory README:

```markdown
# <dir>/ — <одно предложение: за что отвечает>
## Purpose            — зачем существует
## Contents           — таблица файл → ответственность
## Key concepts       — что понять до правок
## Invariants & gotchas — что нельзя ломать (← важнее всего)
## How to extend      — типовая задача по шагам
## Related            — ссылки на смежное
```
